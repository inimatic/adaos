# Endpoint Infrastructure

Status: target architecture and first implementation slice.

This document defines how AdaOS should model ReDevice agents, browser surfaces,
and future device agents as managed endpoints. It complements
[Device Access and Browsers](device-access-and-browsers.md), which describes
access policy for browsers and members, and [Routing](../concepts/routing.md),
which describes current IO and skill routing.

## Core Decision

ReDevice and browser surfaces are not normal AdaOS members with installed
skills. They are endpoint infrastructure.

```text
skills run on hub/member runtimes
endpoints expose services
assignments bind human roles to endpoints
the router delivers commands and streams
scenarios orchestrate skills plus endpoint services
```

This avoids turning an old or damaged phone into a weak hub or a small desktop
runtime. A ReDevice remains a native, low-level endpoint agent that can be
admitted, diagnosed, assigned roles, and controlled through policy-bound
services.

## Vocabulary

- `EndpointRegistry`: durable registry of admitted endpoint records.
- `EndpointManifest`: local endpoint description generated by the agent after
  admission and policy issuance.
- `EndpointPolicy`: hub-issued authorization for services, roles, limits, and
  trust level. It is the source of permissions.
- `EndpointAssignment`: operator-owned role binding, such as assigning one
  physical ReDevice to `demo_display`.
- `EndpointRouter`: service that resolves endpoint roles to concrete endpoint
  services and transport adapters.
- `EndpointCommand`: bounded command sent to an endpoint service.
- `EndpointEvent`: event emitted by an endpoint service or agent lifecycle.
- `EndpointStream`: active data stream from or to an endpoint service.
- `ServiceState`: current state of each endpoint service.
- `EndpointHealth`: heartbeat and health snapshot for the endpoint.
- `ReDeviceAgent`: native-first agent running on an old or constrained device.

Avoid using `Supervisor` for ReDevice state. Use `ACTIVE_ENDPOINT` for the
post-admission active mode.

## Endpoint Lifecycle

The ReDevice lifecycle is:

```text
native ReDevice Agent
  -> local diagnostics
  -> controller-assisted admission
  -> hub-issued endpoint policy
  -> endpoint registry record
  -> endpoint role assignment
  -> active endpoint mode
  -> commands, streams, health, recovery, repeated diagnostics
```

Admission is only the first lifecycle phase. After admission the agent remains
responsible for:

- health reporting and heartbeat;
- reconnect after transport loss;
- local recovery and degraded interaction;
- repeated diagnostics on request;
- service state monitoring;
- policy refresh and revoke/retire handling;
- endpoint events and endpoint streams.

## Endpoint-Only Rule

ReDevice admission must default to endpoint-only. An unreliable old phone,
tablet, or damaged device must not be promoted into an AdaOS hub. The hub owns
subnet admission decisions, endpoint policy issuance, and routing authority.

## Registry Model

`EndpointRegistry` is durable infrastructure state, not pairing-session state.
Pairing and admission records are temporary. An admitted endpoint must remain
visible after a pairing token expires.

Recommended record shape:

```json
{
  "endpoint_id": "endpoint:redevice:tf201-01",
  "kind": "redevice",
  "state": "active",
  "zone": "ru",
  "display_name": "Transformer Prime",
  "trust_level": "limited",
  "manifest_id": "manifest:...",
  "policy_id": "policy:...",
  "admission_session_id": "adm:...",
  "assignments": ["demo_display"],
  "services": {
    "display_endpoint": {"state": "ready"},
    "audio_output_endpoint": {"state": "ready"},
    "button_endpoint": {"state": "ready"}
  },
  "health": {
    "online": true,
    "last_seen_at": "2026-06-01T10:00:00Z",
    "battery": "external_power",
    "connectivity": "reachable"
  }
}
```

The registry should store facts and current state. UI-specific grouping,
operator action availability, and dashboard cards are projections over this
model.

## Endpoint Services

Endpoint services expose capabilities; they do not grant permissions by
themselves. The permission chain is:

```text
hardware capability
  -> endpoint service
  -> endpoint policy
  -> skill permission
  -> runtime confirmation when required
```

MVP ReDevice services:

- `health_endpoint`: heartbeat, battery, connectivity, degraded state.
- `display_endpoint`: show text, image, simple layout, or slideshow.
- `audio_output_endpoint`: play bundled prompt, tone, or transferred audio.
- `camera_qr_endpoint`: scan a QR code for admission and recovery flows.
- `button_endpoint`: hardware volume/button feedback events.

Future services:

- `audio_input_endpoint`: microphone capture or low-rate audio stream.
- `camera_snapshot_endpoint`: still image capture.
- `sensor_endpoint`: GPS, accelerometer, gyroscope, orientation, ambient data.
- `presence_endpoint`: simple motion/presence state.
- `media_endpoint`: bounded video or richer media streaming.

## Assignments

Skills should not target a physical ReDevice directly in normal operation.
They target an endpoint role.

```json
{
  "assignment_id": "assign:demo_display",
  "role": "demo_display",
  "endpoint_id": "endpoint:redevice:tf201-01",
  "service_bindings": {
    "display": "display_endpoint",
    "audio": "audio_output_endpoint"
  },
  "policy_constraints": {
    "max_image_bytes": 524288,
    "allow_camera": false
  }
}
```

This lets an operator replace the physical device without rewriting skills or
scenarios.

## Skill Interface

Skills on hub or members interact with endpoints through AdaOS core services
and SDK helpers. They do not speak to ReDevice transport directly and they do
not use Yjs as the endpoint execution path.

Target SDK surface:

```text
sdk.endpoints.list(filters)
sdk.endpoints.get(endpoint_ref)
sdk.endpoints.assign_role(endpoint_ref, role, bindings)
sdk.endpoints.unassign_role(role)
sdk.endpoints.get_policy(endpoint_ref)
sdk.endpoints.get_health(endpoint_ref)
sdk.endpoints.get_service_state(endpoint_ref)

sdk.endpoint_router.resolve(role, service)
sdk.endpoint_router.send_command(target, command)
sdk.endpoint_router.subscribe(target, stream, options)
sdk.endpoint_router.unsubscribe(subscription_id)
```

The same SDK surface should work when the calling skill runs on the hub or on a
member. A member-hosted skill sends its endpoint request to the hub-owned
endpoint router; the hub applies policy, resolves assignment, chooses the
transport adapter, and records audit evidence.

## Endpoint Commands

Commands are bounded, policy-checked requests with explicit delivery state.

```json
{
  "command_id": "cmd:01H...",
  "target": {
    "role": "demo_display",
    "service": "display_endpoint"
  },
  "type": "display.show_text",
  "payload": {
    "text": "Kitchen timer finished",
    "priority": "normal"
  },
  "constraints": {
    "timeout_ms": 5000,
    "idempotency_key": "timer:123:finished"
  },
  "requested_by": {
    "skill": "timer_skill",
    "node_id": "hub:main"
  }
}
```

The router returns a delivery envelope, not only a boolean:

```json
{
  "command_id": "cmd:01H...",
  "state": "accepted",
  "endpoint_id": "endpoint:redevice:tf201-01",
  "transport": "redevice_poll",
  "accepted_at": "2026-06-01T10:00:01Z"
}
```

Later events update the command:

- `endpoint.command.delivered`
- `endpoint.command.acknowledged`
- `endpoint.command.completed`
- `endpoint.command.failed`
- `endpoint.command.expired`

## Endpoint Events

Endpoint events are low-volume facts from the agent or services:

```json
{
  "event_id": "evt:01H...",
  "endpoint_id": "endpoint:redevice:tf201-01",
  "service": "button_endpoint",
  "type": "button.volume_down.confirmed",
  "phase": "active_endpoint",
  "observed_at": "2026-06-01T10:00:05Z",
  "payload": {
    "sequence": 42
  }
}
```

Events should be normalized by the hub before skills consume them. Raw device
quirks stay in transport adapters or diagnostics.

## Endpoint Streams

Endpoint streams are active subscriptions, not Yjs projections.

Use streams for:

- audio input;
- video or camera frames;
- GPS samples;
- accelerometer, gyroscope, and orientation samples;
- button event tails;
- active service telemetry;
- binary display/content acknowledgements.

The control path is metadata and policy; the data path can be transport
specific.

```json
{
  "subscription_id": "sub:01H...",
  "target": {
    "role": "runner_phone",
    "service": "sensor_endpoint"
  },
  "stream": "sensor.accelerometer",
  "mode": "sampled",
  "options": {
    "sample_hz": 5,
    "max_latency_ms": 1000,
    "duration_ms": 60000
  },
  "consumer": {
    "skill": "motion_skill",
    "node_id": "member:analysis"
  }
}
```

Stream payloads should use bounded frames:

```json
{
  "subscription_id": "sub:01H...",
  "sequence": 17,
  "timestamp": "2026-06-01T10:00:08.120Z",
  "content_type": "application/vnd.adaos.sensor.accelerometer+json",
  "payload": {
    "x": 0.1,
    "y": 9.7,
    "z": 0.2
  }
}
```

For high-bandwidth video, audio, or images, the event bus should carry stream
metadata, references, and lifecycle events. Bulk bytes should use a dedicated
transport or content reference:

- HTTP range/download for small static assets;
- object/content store references for images and audio prompts;
- WebRTC or a future media sidecar for realtime audio/video;
- endpoint polling only for low-rate or legacy fallback modes.

## Content To Endpoint

Display and audio services need both immediate commands and reusable content
references.

Recommended split:

- `display.show_text`: inline small text.
- `display.show_image`: `content_ref` plus fit/background options.
- `display.start_slideshow`: ordered content refs and timing.
- `audio.play_prompt`: stable prompt id from a voice pack.
- `audio.play_content`: transferred or preloaded content ref.

```json
{
  "type": "display.show_image",
  "payload": {
    "content_ref": "content:sha256:...",
    "mime": "image/jpeg",
    "fit": "contain",
    "background": "#000000"
  }
}
```

Legacy devices may only support small images or polling-based content fetch.
Those limits belong in EndpointPolicy and ServiceState.

## Yjs Boundary

Yjs remains a browser/webspace collaborative state and projection mechanism.
It is not the endpoint command bus, stream bus, or source of truth for endpoint
registry records.

Allowed Yjs usage:

- compact operator dashboard projections;
- current selected endpoint/role in a webspace;
- browser-local UI state for Connected Devices panels;
- status cards pointing to endpoint details or streams.

Not allowed as the endpoint interface:

- writing endpoint commands into arbitrary Yjs paths;
- streaming GPS/audio/video through Yjs;
- treating a Yjs document as the durable endpoint registry;
- using browser projection state as the policy source.

## Transport Adapters

The EndpointRouter chooses a transport adapter after policy and assignment
resolution.

Initial adapters can be simple:

- `redevice_poll`: legacy-friendly Android agent polls the hub for pending
  commands and posts events/results.
- `http_callback`: endpoint reachable through direct HTTP.
- `browser_channel`: browser endpoint receives commands through its existing
  live session.

Future adapters:

- WebSocket control channel;
- WebRTC data/media channel;
- local BLE or LAN bridge;
- media sidecar for video/audio streams.

Transport choice is an implementation detail. Skills and NLU should see
roles, services, commands, events, and streams.

## Policy And Trust

Default ReDevice trust should be `low` or `limited`.

Policy should gate:

- which services are enabled;
- which skills may command the endpoint;
- stream direction and bandwidth;
- camera/microphone access;
- local storage;
- maximum content size and retention;
- whether runtime confirmation is required;
- whether degraded mode disables an assignment.

Display commands are lower risk than microphone, camera, or location streams.
Camera, microphone, GPS, and persistent recording should require explicit
policy and visible audit.

## NLU And Scenarios

NLU should classify endpoint-directed phrases as `endpoint_command`, not as
`interface_action`.

Examples:

- "show this on the old tablet" -> `endpoint_command`
- "start slideshow on the kitchen display" -> `endpoint_command`
- "open connected devices panel" -> `interface_action`
- "assign this tablet as demo display" -> `skill_action` or governed
  endpoint assignment action, depending on the owning surface

Scenarios can orchestrate all action planes:

```text
scenario_flow
  -> skill_action
  -> endpoint_command
  -> interface_action for operator review
```

## Connected Devices Skill

The first operator skill should evolve from `redevice_list` into a generic
Connected Devices or Endpoint Registry skill.

It should show:

- members and endpoints separately;
- ReDevice and browser endpoints;
- state, trust, role, heartbeat, and degraded mode;
- manifest, policy, health, diagnostic report, and service state;
- actions: assign role, send test display command, request diagnostics,
  revoke, retire.

The skill is a consumer of EndpointRegistry and EndpointRouter, not the owner
of endpoint infrastructure.

## First Vertical Slice

Recommended M1:

1. Persist ReDevice endpoints in EndpointRegistry after admission.
2. Add `display_endpoint` command polling to the Android ReDevice agent.
3. Add one assignment role: `demo_display`.
4. Add EndpointRouter API for `display.show_text`.
5. Extend Connected Devices skill to assign `demo_display` and send a test
   display command.
6. Add NLU examples for endpoint command preview and dispatch:
   "show status on old tablet".

This proves the architecture without starting with high-bandwidth media,
microphone, GPS, or generic fleet management.
